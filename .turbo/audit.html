02Why joining feels unreliable

The shipped multiplayer is not the parked docs architecture (no Auth/Postgres/RLS). It is Supabase Realtime Broadcast + Presence on public channels: the creator’s tab is the sole authority, and a monotonic seq on full snapshots is the only reconciliation primitive. Five findings compound:

The compounding failure chain

  1. Joiner-ahead-of-host is unrecoverable (CORR-1). Resyncs and corrections never bump seq, so the joiner’s seq <= guard drops them forever — after host storage loss, “New game” divergence, or multi-tab stomps, the joiner goes permanently deaf. ✓ Fixed 06-11 (epoch protocol).
  2. Two tabs in one browser collide (CORR-4). One global selfId + host-config keyed by game id ⇒ your own invite link in a second tab becomes a second host with the same identity; presence sees one client; both wait forever. This silently breaks the most natural way to demo the app. ✓ Fixed 06-11 (per-tab identity + presence-following seats + duplicate-host notice).
  3. Stale “synced” joiner, absent host (CORR-5). A persisted snapshot boots the joiner as synced and movable with nobody listening; intents retry into the void. ✓ Fixed 06-11 (liveSynced gate).
  4. Mid-turn duck snapshots are fatal (CORR-2/3). The host can commit a half-turn, and restoring a duck-phase snapshot crashes placeDuck (empty turns). ✓ Both fixed 06-11 (atomic probe-clone apply + guarded annotation).
  5. No terminal error state (CORR-8). The panel’s “Connection lost — try Resync” branch is unreachable; failures present as infinite “Reconnecting…”. ✓ Fixed 06-11 (terminal error after 6 attempts; Resync reconnects).

Fix class: introduce an epoch (game-instance id) beside seq, bump on every resync/correction, validate wire input, and gate play on live host presence. That one cluster removes the “login feels flaky” symptom — and it is exactly the groundwork the lobby needs.
